## 核心概念

| 概念 | 含义 |
| --- | --- |
| 权威远端 | SiYuan Kernel 中当前真实存在的内容和元数据；它会继续被 GUI、插件、其他客户端和同步任务修改 |
| 远端观察（Observed Remote） | 当前命令经过重复读取验证后接受的一组 Kernel 状态；它是逐文档版本的集合，不是假设出来的全局瞬时快照 |
| 远端跟踪快照（Remote Tracking） | `refs.remote` 保存的最近一次已接受远端观察 |
| 基线（Baseline） | HEAD 所指向、当前本地修改赖以比较的已接受远端观察；稳定且无冲突时与远端跟踪快照相同 |
| 工作副本 | 用户可以自由编辑的本地 Markdown |
| 暂存状态 | 用户通过 `add` 选入下一次提交的完整候选状态 |
| 提交意图 | `commit` 冻结的目标状态；之后继续编辑工作副本不会改变它 |
| Patch | 从基线到目标状态的一次完整变更；一个 Commit 对应一个 Patch，Patch 内按文档组织操作 |
| 同步事务 | 一次由 pull 或 push 发起、尝试从已确认基线推进到新基线的完整状态转换 |
| 事务记录 | 同步进行期间保存的当前阶段、前置条件、远端响应和回读证据，用于中断后继续判断 |
| 规范结果 | Kernel 接受写入、为新增 Block 分配 SiYuan Block ID 并规范化内容后，对本次受影响文档重新读取到的状态 |
| 内容冲突 | Base、Local 和本次 Observed Remote 无法自动合并成唯一结果 |
| 事务不确定性 | 无法证明远端副作用或本地物化是否已安全完成 |

这里最重要的区分是：

- 权威远端是持续变化的实时系统；远端观察是某次命令通过重复读取接受的版本集合；远端跟踪快照是本地最后接受的观察。
- 基线和目标状态描述“是什么样”。
- Patch 描述“如何从一个状态走到另一个状态”。
- 同步事务界定一次完整的状态转换，事务记录描述“转换已经走到哪里、哪些事实已经得到证明”。
- 内容冲突表示目标无法自动合并，事务不确定性表示执行结果无法得到证明。

## 设计决策

### 参考 Git 的状态与存储模型

`siyuan-worktree` 不复制 Git 的全部版本控制能力，只借用适合本地同步的几个基础思想：

| Git 思想 | 在本项目中的作用 |
| --- | --- |
| Working Tree、Index、HEAD | 区分正在编辑、已经选择和等待 push 的内容状态 |
| 完整快照 | 直接描述基线和目标，不依赖此前执行过哪些修改步骤 |
| 内容寻址对象 | 让快照、Patch 和事务记录不可变、可校验并复用未变化内容 |
| 原子引用更新 | 让多个内容状态面或事务阶段一起生效，避免只更新一部分 |
| diff/apply | 从基线和目标生成 Patch，再通过 Kernel 安全执行转换 |

快照、Patch 和事务记录承担不同职责：快照描述某个边界上的完整状态，Patch 描述从基线到目标的变化，事务记录保存 Patch 已经执行到哪里以及有哪些证据。状态判断以快照为准；Patch 可以从两个快照重新生成和校验；事务记录只服务于当前 pull 或 push 的恢复。

这些思想通过 `pull`、`add`、`commit`、`push`、`status`、`diff` 和 `reset` 等接近 Git 的命令暴露给用户，但各命令操作的是 SiYuan 同步状态，不承诺 Git 的分支和长期历史语义。

### 充分利用 SiYuan 已有的原生能力

尽量使用 SiYuan 已经提供的能力，而不是直接操作内部数据文件：

| SiYuan 能力 | 在模型中的角色 |
| --- | --- |
| 文档和块读取 | 建立 Observed Remote、执行前置检查和写后验证 |
| Transaction queue flush | 在读取前等待已经排队的 Kernel transaction 完成，作为读取屏障 |
| 文档历史 | push 写入前建立单文档人工恢复点 |
| 块级写接口 | 执行 Patch 中的插入、更新和删除 |
| Transaction 返回信息 | 提供 Kernel 已处理操作的证据，以及新创建 Block 的 SiYuan Block ID |
| 规范化回读 | 决定受影响文档在 push 后进入本地基线的内容，而不是直接相信请求内容 |
| 数据仓库快照 | 可选的批量操作检查点、审计和人工恢复层 |

这些能力提供恢复和验证证据，但不能让本地文件系统与 Kernel 获得一个共同的原子事务。queue flush 返回后新的写入仍可立即进入；标准 Kernel 也没有对外写租约或带 expected revision 的条件写接口。状态机仍必须处理响应丢失、外部并发写入和部分文档成功。

第一阶段把 SiYuan 属性视为只读元数据。本地内容更新必须保留远端已有属性；如果用户在本地修改已有块属性，提交应被拒绝。push 在内容 mutation 前后都会验证这些属性，发现差异时停止，不使用无条件属性写入把旧值覆盖回去。未来若支持属性编辑，应把它建模为独立、可校验的操作，而不是内容更新的隐式副作用。

### 本地工作树采用单写者假设

当前模型假设同一个本地工作树在命令执行期间只有 `siyuan-worktree` 会写入。工作树锁负责串行化本项目的 CLI 进程；用户应避免在 pull/push 正在物化文件时让编辑器自动保存，也不应让网盘同步、文件监控脚本或其他程序同时改写映射的 Markdown。

因此，本地文件并发写入不进入跨系统冲突模型，也不为普通文件系统设计租约、文件 CAS 或多写者合并协议。实现仍保留写入前内容哈希复检作为低成本保护：如果发现文件在事务期间变化，命令会停止而不是覆盖；这项检查是防误用的保护栏，不代表支持通用的本地多写者场景。

### 只同步，没有进一步的版本控制

`siyuan-worktree` 是同步工作树，而不是与 SiYuan 并列的第二套版本控制系统。

存在两个前提：

- SiYuan Kernel 是长期权威数据源。
- SiYuan 已经提供文档历史和数据仓库快照用于长期恢复。

所以本地只需要最近一次确认的基线，以及当前尚未完成的同步事务，不需要长期提交图，长期提交图还会引入分支、合并、变基、历史保留和两套历史分叉等额外语义。另外如果本地历史和 Kernel 历史都被视为权威，冲突将从“同步两个状态”升级成“协调两个版本控制系统”。

```text
稳定状态
  最新 Baseline

编辑和提交期间
  Baseline + Staged + Commit Intent + Patch

同步期间
  上述状态 + Transaction Evidence + Canonical Result

同步完成
  新 Baseline
```

旧基线、已完成提交和事务证据在不再被当前状态引用后可以回收。长期恢复交给 SiYuan 历史，不在本地形成持续增长的 Commit 图。

同时本地 Commit 仍然有价值，但它只是一次 push 的短期边界：冻结用户已经批准的目标状态和 Patch。push 成功后，受影响文档的 Kernel 规范结果进入新基线，旧提交和事务证据即可退出活动状态；这不表示对整个 Kernel 做了一次全局快照。

## 状态机设计

### 内容状态机

`siyuan-worktree` 的主状态机参考 Git 的 Working Tree、Index 和 HEAD 模型，用来描述用户内容当前处于什么状态。它不是一个单独的 `clean/dirty/conflict` 枚举，而是由多个内容状态面之间的关系共同决定。

Git 的类比只用于区分各状态面的职责，并非完全一一对应；`siyuan-worktree` 还需要表示最近确认的远端基线和当前观察到的 Kernel 状态：

| 状态面 | Git 类比 | 表示什么 |
| --- | --- | --- |
| Observed Remote（远端观察） | 一次 fetch 结果 | 当前命令通过重复读取验证后接受的 Kernel 版本集合 |
| Remote Tracking（远端跟踪快照） | remote-tracking ref | `refs.remote` 保存的最近一次已接受远端观察 |
| Baseline（基线） | 当前共同起点 | HEAD 所指向、当前本地修改赖以比较的远端观察 |
| Working | Working Tree | 用户正在编辑的工作副本 |
| Staged | Index | 用户已经选择的下一次提交候选状态 |
| Committed | HEAD 中的本地提交 | 已冻结、等待 push 的目标状态；不存在待 push 提交时为空 |

这些状态面之间的比较产生用户看到的状态：

| 条件 | 含义 |
| --- | --- |
| Working 与 Baseline 相同 | 本地没有未同步编辑 |
| Working 与 Staged 不同 | `add` 后又发生了编辑 |
| Staged 与 Baseline 不同 | 存在已暂存修改 |
| Committed 存在 | 存在等待 push 的提交意图 |
| Observed Remote 与 Remote Tracking 不同 | 本次接受的远端观察与上次跟踪结果不同 |
| Observed Remote 与 Baseline 不同 | 本次接受的远端观察已偏离当前共同起点 |
| 本地和 Observed Remote 都偏离 Baseline | 需要三方比较，可能合并也可能冲突 |

#### 三方合并与内容冲突

内容合并以三个状态为基础：

```text
Base   最近共同确认的基线
Local  本地希望保留或提交的状态
Remote 本节中专指当前命令接受的 Observed Remote
```

三方比较产生以下内容状态转换：

| Local 相对 Base | Remote 相对 Base | 结果 |
| --- | --- | --- |
| 未变化 | 未变化 | 保持已同步 |
| 未变化 | 有变化 | 接受 Remote |
| 有变化 | 未变化 | 保留 Local |
| 有变化 | 有变化且互不影响 | 自动合并 |
| 有变化 | 有变化且相互重叠 | 产生内容冲突 |
| 与 Remote 已经相同 | 有变化 | 直接收敛到共同结果 |

内容冲突不只意味着文本不同。以下情况都不能自动生成唯一目标状态：

- Local 和 Remote 修改了同一个逻辑块，无法证明修改兼容。
- 目标块、插入锚点或预期旧内容已经变化。
- 远端对象消失，但无法区分删除和映射范围变化。

解决内容冲突的本质是重新选择目标状态：保留 Local、接受 Remote，或者基于 Base、Local、Remote 手工合并出新的 Local。解决后重新经过 `add -> commit -> push`，让新目标再次接受完整校验；冲突处理不直接绕过状态机写入 Kernel。

### 同步事务状态机

同步事务状态机是内容状态机的支撑层，而不是与内容状态面并列的另一组业务状态。内容状态机决定“当前是什么状态、目标要变成什么状态”；事务状态机只负责让一次 pull 或 push 安全地完成这次转换。

事务执行期间可能已经产生部分远端副作用，但在验证和本地物化完成前，这些进度不会被宣布为新的稳定内容状态。事务成功结束后只留下新的基线，阶段和恢复证据随之退出活动状态；如果因内容冲突或事务不确定性而停止，事务记录会继续保留，供恢复或人工处理。

#### Pull 事务

pull 的目标是把远端新事实合并到本地，同时不覆盖用户尚未同步的编辑。它在语义上与 Git 一致，可以拆成两个连续步骤：

1. **fetch**：只读取 Kernel，建立本次接受的 Observed Remote 版本集合。
2. **merge/rebase**：以 Baseline 为共同起点，比较 Local 与 Observed Remote，把远端变化合并进工作副本，或者把仍然有效的本地变化重放到新的 Remote Tracking 上。

```text
fetch
  Kernel -> Observed Remote

merge/rebase
  Baseline / Local / Observed Remote 三方比较
  -> 选择快进、保留本地、自动合并或内容冲突
  -> 更新工作副本
  -> 验证本地结果
  -> 将 Observed Remote 确认为新 Remote Tracking
  -> 无冲突时将 Baseline 推进到同一远端观察
```

具体的三方判断和冲突解决属于内容状态机。Pull 事务负责完成 fetch、执行内容状态机产生的 merge/rebase 决定，并可靠地物化结果。

事务开始时先稳定读取并冻结 Working 状态。fetch 先通过读取屏障等待调用前已经排队的 Kernel transaction 完成；随后连续采集整个映射范围。每一轮先读取 notebook/doc inventory，再逐篇采集块关系、Kramdown 和属性，并在单篇采集前后复核块结构，最后再次确认 inventory 没有变化。只有连续两轮得到相同的 inventory 和完整文档版本集合，才接受本次远端观察；持续编辑会让 pull 重试，重试耗尽后安全停止。

Observed Remote 是“文档 ID 到已验证文档版本”的版本向量，不承诺所有文档来自 Kernel 的同一个全局时刻。完整版本向量必须连续两轮相同，每篇文档在单轮采集中的结构也必须前后一致；这能发现先读完的文档在后续扫描期间发生的大多数变化，但仍只是乐观稳定性检查，不是线性一致性证明，不能排除 ABA，也不能阻止最后一次读取之后立即发生的新写入。

fetch 和三方比较完成后，先持久化完整的本地物化计划，再逐篇更新 Markdown、metadata 和冲突材料；每完成一篇就记录进度。物化前还要验证文件仍与事务开始时的内容一致，或者已经等于本事务的目标内容。全部文件和本地状态完成后，才原子确认新的 Remote Tracking；没有冲突时再同时确认新的 Baseline。

pull 只读取 Kernel，不产生远端写入。它的风险主要在于多文件工作副本更新过程中发生中断或新的编辑，因此完成前不能把半套本地文件宣告为新基线。

如果存在尚未收敛的 push，pull 不开始一笔普通工作副本同步，而是建立新的 Observed Remote，并把它与 PushOperationState 中的已完成文档前缀、in-flight 操作、receipt 和剩余意图对齐。已经达到目标状态的 Update/Delete/Insert 记为完成，仍处于可信前态的操作记为未执行，剩余 Patch 可以在远端无关变化上安全重放时保持可继续；无法证明时产生冲突。该过程只读取 Kernel，不重复写入，也不回滚已经验证的文档。

#### Push 事务

push 的目标是把已经提交的目标状态安全应用到 Kernel，并以受影响文档的规范化回读结果更新本地基线；该结果不表示整个 Kernel 在同一全局时刻的快照。

```text
准备
  -> 检查基线和本地前置条件
  -> 建立恢复点
  -> 应用 Patch
  -> 保存远端响应证据
  -> 回读并验证 Kernel 结果
  -> 形成规范结果
  -> 更新本地工作副本
  -> 原子确认新 Baseline
```

各阶段的语义是：

| 阶段 | 必须证明的事实 |
| --- | --- |
| 准备 | 提交意图完整，目标与 Patch 匹配，本地没有会被回读覆盖的新编辑 |
| 前置检查 | 在创建历史或执行任何 mutation 前，所有待修改文档都已完成稳定预检；Kernel 仍符合这笔提交所依赖的前置条件，或者变化可以被明确处理 |
| 建立恢复点 | 高影响写入前已经请求 SiYuan 保存可供人工恢复的历史 |
| 应用 | 每个远端操作都按既定顺序执行，并记录其开始和返回证据 |
| 验证 | 重新读取 Kernel，证明实际结果与操作目标一致 |
| 规范化 | 接受 Kernel 为新增 Block 分配的 SiYuan Block ID、规范化内容和保留的元数据 |
| 本地物化 | 用规范结果更新 Markdown 和本地映射信息 |
| 确认 | 只有远端验证和本地物化都完成后，规范结果才成为新基线 |

Kernel 返回“成功”不等于同步完成。Kernel 可能为新增 Block 分配 SiYuan Block ID、转换内容或保留额外元数据，所以必须回读；push 的最终结果是回读得到的规范状态，而不是发送前的本地文本。

预检会先覆盖整个 Commit 中的待修改文档，避免第一篇文档已经写入后才发现后续文档从一开始就存在冲突。每个操作执行前还会再次稳定读取其目标和结构前置条件。不过标准 Kernel 的 mutation 是无条件写入，外部客户端仍可能恰好在“最后一次校验完成”和“Kernel 实际执行写入”之间修改同一对象；客户端只能缩小并检测这个窗口，不能把它消除。

多文档 push 使用与 Git rebase/cherry-pick sequencer 相似的已完成前缀：每篇文档只有在全部块操作完成并经过 canonical 回读后才推进 `AppliedDocuments`。后续文档失败不会撤销这个前缀；`status` 显示事务阶段、已完成前缀、当前文档和下一条恢复命令，下一次 pull 负责 fetch 和对账，`push --continue` 跳过已经证明完成的文档，只执行剩余部分。已经产生远端效果后不存在普通 `abort`，因为本地状态机不能假装 Kernel 写入没有发生。

#### 中断恢复原则

本地状态更新可以原子化，但下面三个动作不能组成跨系统原子事务：

```text
本地记录“即将写入”
  -> 调用 SiYuan Kernel
  -> 本地记录“Kernel 已返回”
```

进程可能在任意两个动作之间退出。恢复模型因此遵循四条原则。

事务阶段只向前推进。错误和事务不确定性附着在当前阶段上，不应让阶段倒退或假装之前的外部效果没有发生。

**先记录意图，再产生副作用**

调用 Kernel 前先持久化本地 `operationId`、操作目标和前置条件。这样中断后至少知道可能发生了什么。

**响应只是证据之一**

收到 Kernel 响应后保存响应，但仍要回读目标对象。响应证明请求被处理过，回读证明当前远端实际是什么。

**只推进已经证明的阶段**

每个阶段只能在所需证据完整后向前推进。重启后从最后一个已证明阶段继续，而不是重新执行整笔 push。

**不确定时不重试破坏性操作**

如果无法区分“请求没有执行”和“请求已经执行但响应丢失”，盲目重试可能造成重复插入或错误删除。此时先保留事务记录，由下一次 `pull` 重新读取 SiYuan 并对账，而不是猜测或直接重试。

因此，状态机保存的核心不是调用日志，而是恢复判断需要的证据：

- 操作前观察到的状态。
- 准备执行的不可变意图。
- Kernel 返回的凭据。
- 操作后的回读结果。
- 已完成本地更新的范围。

#### 事务不确定性

事务不确定性不是 Base、Local、Observed Remote 之间的内容合并冲突，而是状态机无法证明一次同步能否安全继续。典型情况包括：

- Kernel 可能已经执行操作，但响应或本地 receipt 没有持久化。
- 同步期间工作副本又被编辑，继续物化会覆盖新内容。
- 外部写入发生在前置检查之后，现有证据无法确定本次操作的实际结果。

发生这类情况时，事务阶段保持在最后一个已证明的位置，并保留事务记录。系统不会把不确定结果推进为新基线，也不会自动重复可能造成二次副作用的操作。

下一次 `pull` 会建立新的 Observed Remote，并对当前 in-flight 操作产生三种基础判定：

| 对账结果 | 后续处理 |
| --- | --- |
| 远端已经是目标状态 | 记录操作已完成并推进已完成前缀；Update 使用目标 Kramdown，Delete 使用 Block 不存在，Insert 使用锚点、内容和返回的 Block ID 证明 |
| 远端仍是可信写入前状态 | 清除 in-flight 标记，保留剩余意图，允许 `push --continue` 安全重试 |
| 远端既不是目标结果，也不是可信前态 | 进入内容冲突，保留事务记录和三方材料 |

如果没有 in-flight 操作，pull 仍会验证已完成操作和当前剩余 DocumentPatch。远端只修改了 Patch 不涉及的 Block 时，剩余意图无需改写即可在新远端上继续，最终 canonical 文档会同时包含双方变化；同一目标发生不兼容变化时保留冲突。这里仍然遵循 `fetch + merge/rebase`，不会在存在待完成 push 时执行新的 Kernel 写入，也不会直接覆盖工作副本。

## 状态机能力边界

以下约束不是增加更多本地状态就能消除的。状态机只能检测、保存证据并安全停止，不能提供底层系统本身没有的保证：

- SiYuan Kernel 不提供跨文档原子事务。多文档 push 可能只完成一部分，状态机可以精确记录完成范围，但不能把已经发生的远端写入整体回滚。
- 稳定观察协议只接受连续两轮相同的完整映射版本集合，并要求每轮 inventory 和单篇文档结构通过前后复核；但它不是 Kernel 锁或线性一致读事务，不能排除 ABA，也不能阻止最后一次读取后立即发生的新写入。
- SiYuan Kernel 没有供外部客户端持有的临时全局写锁，也没有带 expected hash/revision 的条件 mutation。`setEditorReadOnly` 只影响官方编辑器 UI，`flushTransaction` 只是读取屏障，`/api/transactions` 只在执行单笔 Transaction 时串行 mutation。因此 push 可以在写入前稳定预检、在写入后回读并在不确定时停止，但仍无法从客户端彻底消除最后的 check-then-write 竞争窗口。
- 如果一次 Insert 请求已经发出但响应丢失，在下次 `pull` 之前，用户又通过 SiYuan 或其他客户端在相同的前后块之间插入了内容，那么远端会同时出现多段来源不同的新增内容。此时 `pull` 可能无法确认其中哪一部分来自上次 push。为避免重复插入或误处理其他客户端的内容，程序会停止自动处理并保留冲突，等待用户确认。
